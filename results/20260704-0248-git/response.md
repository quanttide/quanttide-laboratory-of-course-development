# Git 课程蓝图

大多数人学 Git 的方式是这样的：先装个软件，然后打开终端，开始背命令。

`git init` 是初始化仓库，`git add` 是把文件加到暂存区，`git commit` 是提交，`git push` 是推送到远程。每一条命令都配一个解释，像背单词一样。

然后他们很快就会遇到一个困惑：为什么 Git 要搞一个"暂存区"出来？为什么不能直接提交？为什么提交到本地了还要再推一次？为什么远程仓库叫 origin？

这些问题不是记性不好——是教学的起点错了。

## 从 Linus 的愤怒开始

故事要从 2005 年讲起。

当时 Linux 内核是用 BitKeeper 管代码的。BitKeeper 是个商业软件，给 Linux 社区免费用了，但后来因为一些纠纷收回了授权。Linus Torvalds 很生气，决定自己写一个版本管理工具。

他当时面临一个具体的问题：Linux 内核有上千个贡献者，遍布全球各地。这些人有的在 MIT，有的在芬兰，有的在自己家里。他们不可能连同一个中央服务器来工作——且不说服务器扛不住，那个年代的网络速度也不允许。

所以 Linus 对新工具的要求很简单：每个人本地都要有完整的代码仓库，不需要联网也能工作。这就是"分布式"的由来。

你把这个故事讲完，再回过头来看 Git，一切就顺了。

## 三件事，分三次讲

### 第一次：不讲远程

第一次上课，让大家就在自己的电脑上玩。

先 `git init`。然后随便写一个文件，`git add` 把它加进去，`git commit -m "first commit"`。然后再改一下，再 add，再 commit。就这么循环。

这时候一定会有人问：为什么要 add 一下再 commit？不能直接 commit 吗？

这就是个好问题，你得接着讲故事。

"假设你正在写一个报告，你同时改了两个部分。第一部分改完了，第二部分还没改完。但是你老板现在就要看第一部分。你怎么办？"

暂存区就是干这个用的。它让你分批次提交。`git add` 就是在跟 Git 说："这几个改好的，先打包。"`git commit` 是把打好的包存起来。没 add 的改动还在工作区，不着急。

这个道理一旦讲通，后面的东西就是顺水推舟。

### 第二次：分支必须单独讲

很多人学 Git 学崩溃，是因为把分支和本地操作混在一起学了。这是教材的问题。

分支就是一个指针。创建分支就是新建一个指针，切换分支就是移动指针。它不复制文件，所以快得离谱。就这么简单。

但为什么要有分支？不是 Git 发明了分支，是软件开发天然就需要分支——你在修 Bug 的时候，新功能开发也不能停。让 Bug 修复和新功能各自待在自己的分支上，互不干扰，修完了再合并回来。

你甚至可以拿我们自己的实践举例：我们所有人都在一条主干分支上工作，这就是所谓的主干开发模式（trunk-based）。好处是简单，坏处是高度依赖自动化——所以我们才造那么多工具。

### 第三次：才是远程

有了本地仓库，有了分支，这时候再讲远程。

远程是什么？就是"另一个仓库"。推送（push）就是"把我的提交复制过去"，拉取（pull）就是"把对方的提交复制过来，顺便合并"。这个理解建立之后，`origin`、`upstream` 这些名字就只是地址的别名而已了。

## 总结

你发现没有，整个 Git 的教学逻辑就是一条线：

> 先理解问题 → 再理解设计 → 最后操作

这条线就是课程蓝图。

你不用去背 Git 的命令，因为你已经知道 Linus 为什么要写这个东西，知道他要解决什么问题，知道他的设计思路是什么。剩下的所有命令，都是这个思路的自然延伸。

蓝图不是一个可以反复使用的教案——它是你做每一门课之前，都要重新经历一遍的追问。这一轮是 Git，下一轮是 Docker，再下一轮是大数据。每一次你都回到原点，问同一个问题：

**这个东西是怎么来的？造它的人当时在想什么？**
