# Git 课程蓝图

## 1. 起源

Git 诞生于 2005 年，由 Linus Torvalds 为 Linux 内核开发而创造。

当时的背景是：Linux 内核团队使用 BitKeeper 做版本管理，但 BitKeeper 收回了免费授权。Linus 需要一个新的版本控制系统，但市面上的工具（CVS、SVN、Perforce）都无法满足他的需求：

- **CVS/SVN** 是中心化的——服务器宕机就无法工作，且分支操作极其笨重
- **Perforce** 虽然快，但同样依赖中央服务器，且是闭源商业软件
- 内核开发是高度分布式的——上千名贡献者遍布全球，每个人都需要独立的完整工作环境

Linus 对版本控制系统的要求非常明确：
1. **分布式**——每个人本地都有完整仓库，不依赖中央服务器
2. **速度极快**——内核代码量巨大，慢的工具无法接受
3. **分支和合并必须轻量且可靠**——内核开发重度依赖分支管理
4. **数据完整性**——绝不能有数据损坏，CRC 校验是基础

Git 只用了两周就写出来了——一个专为"大规模分布式协作"而生的版本控制工具。

## 2. 核心对比：分布式 vs 中心化

| | SVN（中心化） | Git（分布式） |
|---|---|---|
| 仓库位置 | 只有服务器有完整仓库 | 每个人本地都有完整仓库 |
| 离线工作 | 不行，必须连服务器 | 完全离线，本地即可完成所有操作 |
| 分支成本 | 高——分支就是服务器上的目录拷贝 | 低——分支只是一个 41 字节的指针 |
| 协作模式 | 所有人对着同一个中央仓库操作 | 每个人有自己的仓库副本，通过 push/pull 交换 |

这个对比框架是所有 Git 教学的第一块基石。不理解分布式的设计意图，就无法理解 Git 为什么这么"麻烦"——多出来的暂存区、本地提交再推送、看似绕弯子的分支机制，全都是在为分布式服务。

## 3. 关键概念（按逻辑顺序）

```
工作区（你正在编辑的文件）
    │ git add ↓
暂存区（准备打包的快照）
    │ git commit ↓
本地仓库（你的个人历史记录）
    │ git push ↓
远程仓库（团队共享的历史记录）
```

反向路径：
```
远程仓库
    │ git fetch / git pull ↓
本地仓库
    │ git checkout / git merge ↓
工作区
```

### 每个概念的背后逻辑

**暂存区（Index/Staging Area）**：Git 最独特也最让新手困惑的设计。为什么需要它？

想象你改了两个文件，一个解决了 Bug A，一个解决了 Bug B。你不希望它们出现在同一次提交里——因为两个 Bug 是独立的问题，应该分属不同的提交。暂存区让你可以分批次选择性地提交，`git add bugA_fix` 打包一个，`git add bugB_fix` 再打包一个。这是 SVN 做不到的精细控制。

**分支（Branch）**：在 Git 里，分支只是一个指向某次提交的指针（41 字节的 SHA 引用）。创建分支就是新建一个指针，切换分支就是移动一下指针。这也是为什么 Git 的分支操作快得离谱——它不复制文件，只移动指针。

**远程（Remote）**：因为每个人本地都有完整仓库，所以"远程"不是中心服务器，而是"另一个仓库"。"推送"就是"把我本地的提交复制到那个仓库"，"拉取"就是"把那个仓库的提交复制到我本地"。

## 4. 教学流程

### 第一讲：先理解问题

不讲命令，先讲故事：

- 你和同事一起改一个文件，怎么避免覆盖彼此的工作？
- 试着手动维护版本：文件命名为 `v1`、`v2`、`最终版`、`最终版2`——显然不可行
- 这就是版本控制要解决的问题

引出核心对比：中心化 vs 分布式。让大家理解分布式才是 Git 的灵魂。

### 第二讲：本地三区模型

不讲远程，只讲本地。让每个人在自己的电脑上操作：

1. `git init` 创建一个仓库
2. 编辑文件 → `git add` → `git commit` 这个循环
3. 理解暂存区的意义——故意改两个无关的文件，分两次 add，看能不能分别提交

目标：让大家形成"工作区→暂存区→本地仓库"的肌肉记忆。

### 第三讲：分支

这一讲必须独立——**绝不要和本地三区操作混在一起讲**。

从问题入手："你们有没有遇到过同时要修 Bug 和新功能的场景？怎么管理？"
引出分支的概念，然后实操：
1. 创建分支、切换分支
2. 在不同分支上独立工作
3. 合并分支
4. 处理最简单的合并冲突

### 第四讲：远程协作

有了本地仓库和分支的基础，再引入远程：
1. 远程是什么——另一个仓库地址
2. `git remote add`、`git push`、`git pull`、`git fetch`
3. fetch 和 pull 的区别：fetch 只是下载，pull 是下载并合并
4. 多人协作场景：推拉冲突怎么处理

## 5. 常见误区及化解

| 误区 | 化解方法 |
|------|---------|
| "Git 命令太多，记不住" | 别背命令，理解三区模型后命令就自然推导出来了：你要把文件从哪移到哪，就用对应的命令 |
| "暂存区太麻烦，为什么不能直接提交？" | 用两个独立修改的对比实验说明——没有暂存区，你就只能一股脑提交所有改动 |
| "分支和提交搞混了" | 分开两讲，先建立三区肌肉记忆，再讲分支。两个概念之间没有直接操作上的交叉 |
| "push 被拒绝了怎么办？" | 先理解远程比你多了一些提交（别人推的），你需要先 pull 合并，再 push。本质上是对"分布式"的加深理解 |
| "merge 冲突好可怕" | 冲突不是错误，是 Git 在说"我搞不清你要哪个版本"。解决冲突就是开个会，商量一下 |

## 总结

Git 不复杂，复杂的是"在没有理解设计意图的情况下硬背命令"。课程蓝图的起点始终是那个问题：**Linus 在 2005 年面对的是什么问题，他为什么选择了这条路？** 理解了这一点，所有命令都只是这个核心决策的自然延伸。
