# Git 课程蓝图

## 核心教学思路

新手学 Git 最大的困惑是什么？

不是记不住命令，是觉得 Git 很"绕"——

"我在本地提交了一次，为什么还要再 push 一次？这两个不都是保存吗？有什么区别？"

这个问题是 Git 教学的钥匙。所有讲不清 Git 的人，都是在这个问题上绕过去了。

回到设计源头。要解释这个问题，你得先讲清楚一个对比。

## 第一步：SVN 是怎么做的

在 Git 出现之前，最主流的版本管理工具叫 SVN。SVN 是**中心化**的。

什么意思？就是有一台中央服务器，所有人的代码都存在那台服务器上。你想提交代码，你必须先连上那台服务器，在服务器上直接操作。你离了服务器，什么都干不了——不能提交，不能看历史记录，不能对比版本。

这就是"集中式"：所有操作都集中在服务器上完成。

## 第二步：Git 是反过来的

Linus 写 Git 的时候，他面对的场景是这样的：Linux 内核有上千个贡献者，这些人分布在全球各地——有人在美国，有人在欧洲，有人在澳大利亚。他们不可能所有人都连同一台服务器工作，那个年代的网络也不支持。

所以他做了一个决定：**每个人本地都有一个完整的仓库**。

这意味着什么？意味着你不联网也能提交代码，你本地就有一份完整的历史记录，你可以在自己的机器上做任何操作——创建分支、查看历史、对比版本、回退修改。远程有没有服务器，跟你没有关系。

这就是"分布式"：人人都有完整的一份。

## 第三步：回到那个困惑

所以现在你再看这个问题——"为什么还要再 push 一次？"

答案就清楚了：

- `git commit` 是把代码提交到你**本地**的仓库——这是对自己负责
- `git push` 是把本地仓库的提交**复制**到远程仓库——这是对团队负责

两次操作，两个不同的目标。一个是把自己的工作保存起来，一个是把你的工作和别人共享。

SVN 把这两件事混在一起了——你提交就是提交到中央服务器，没有"先本地保存"这一步。Git 把它们拆开了。

这不是 Git 设计得"绕"，而是 Git 给了你更多的控制权。你先在自己的世界里自由操作，准备好了再发布给别人。

## 第四步：那分支呢？

理解了分布式和中心化的区别之后，分支就很好讲了。

SVN 的分支是贵的——因为在服务器上拷贝一份代码很重。Git 的分支是便宜的——因为分支就是一个指针，就在你本地，创建和切换都不需要联网。所以 Git 社区鼓励你大量用分支，这是两种不同的开发文化。

反过来，你也能理解为什么我们现在的实践是主干分支了——我们所有人都在一条主干上开发，用得跟 SVN 一样。但 Git 给了你选择的权利，你愿意的话随时可以切到 feature branch 模式。

## 总结

整个 Git 课程蓝图就围绕一个核心问题展开：**为什么本地 commit 之后还要 push？**

从这个问题出发，你必须去讲分布式和集中式的区别，你必须回到 Git 被设计出来的原始场景。这个设计源头讲清楚了，后面的所有概念——暂存区、分支、远程——都变成了这个核心决策的自然延伸，不需要死记硬背。
