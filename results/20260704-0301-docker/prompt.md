请为 "docker" 设计一份课程蓝图框架。

## 背景

课程蓝图不是教什么，而是为什么这样教。我们遵循追本溯源——回到设计源头，理解它解决了什么问题。

## 要求

找到一个初学者在使用 docker 时**最具体的操作困惑**——比如"为什么做完 X 还要做 Y？"这种让人觉得"多此一举"的问题。从这个困惑出发，回到设计源头讲清楚它为什么被设计成这样。

输出框架，不是讲课稿。包括：

- **核心困惑**：具体是哪个操作让人觉得"绕"
- **设计源头**：回到什么场景/问题来解释这个困惑
- **对比框架**：用什么对比来帮助理解（如分布式 vs 集中式）
- **课时组织**：分几节课，每节课解决什么问题

不要展开成详细的讲课内容，框架即可。
